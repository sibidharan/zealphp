# ZealPHP v0.3.0 Roadmap — Closing the Half-Cooked Surface

**Date:** 2026-05-23
**Status:** Decided after the v0.2.41 audit; framework owner approved scope.
**Intent:** Take the framework from "production-ready for HTTP + WS + Store + shared state" to **complete full-stack** for everything except the DB layer (deferred until we decide on adoption strategy).
**Driving principle:** Close every half-cooked item we identified in the audit; ship in two phases.

---

## Phase 1 (v0.3.0) — close the half-cooked + most-frequently-asked gaps

Ordered roughly by impact × effort ratio.

### P1.1 — WebSocket rooms / channel taxonomy (Phoenix-Channels-style) ✅ SHIPPED (`8dae575` + `2fe6bd2`)

**State today:** `WSRouter::onRoom()`, `WSRouter::broadcast()` exist as thin pub/sub helpers. No membership tracking, no presence, no per-room state.

**Target:** First-class room abstraction on top of the existing Redis fabric:

```php
$room = WSRouter::room('chat:42');
$room->join($clientId);            // SADD into the room's member set; broadcast join event
$room->leave($clientId);           // SREM; broadcast leave event
$room->members(): array            // SMEMBERS — current cluster-wide roster
$room->size(): int                 // SCARD
$room->push(string $payload);      // fan out to every connected member across the cluster
$room->onMessage(callable $fn);    // register an inbound handler
$room->onPresenceChange(callable $fn);  // join/leave events fire this
```

Membership state lives in `ws_room:{name}` SET (cluster-wide, in Redis), with per-server local fd caches for the actual push path. Presence diffs delivered via pub/sub. ~250 LOC + tests + a `learn/websocket-rooms` lesson.

**API design notes:**
- Same FD-coherence + `conn_id` nonce semantics as WSRouter::sendToClient (C1 carries over).
- Room state is OPT-IN — apps not using rooms pay zero cost.
- Reconnect: client re-joins on reconnect; framework doesn't auto-rejoin.

### P1.2 — Reliable queues with consumer ergonomics

**State today:** `Store::publishReliable` + `App::onReliableMessage` use Redis Streams with consumer groups. At-least-once is there. Retry counts, DLQ, scheduled-enqueue, in-flight visibility — all missing.

**Target:** A `Queue` class that wraps Streams with the operational features queue-libraries ship:

```php
$queue = Queue::for('orders');
$queue->push(array $payload, ?int $delaySec = null): string;       // scheduled push via ZSET timer
$queue->stats(): array  // {enqueued, in_flight, dead_lettered, ack_count, mean_runtime_ms}

App::onJob('orders', function (array $payload, JobContext $ctx): bool {
    if ($ctx->attempt > 5) {
        $ctx->dlq();             // route to orders:dlq
        return true;             // ack the original
    }
    return processOrder($payload);
});
```

`JobContext` carries: `attempt` (count from XPENDING idle history), `id`, `enqueuedAt`, `dlq()`, `retry(?int $delaySec)`, `extend()` (heartbeat the in-flight claim for long jobs). Built on `XAUTOCLAIM` (we already have this).

DLQ is a separate stream `{name}:dlq`. Scheduled enqueue uses a sorted-set `{name}:scheduled` + a per-cluster mover coroutine that promotes due jobs to the live stream. ~400 LOC + tests + a `learn/queues` lesson.

### P1.3 — Proper auth flows (OAuth, JWT, session-cookie)

**State today:** `App::authChecker/adminChecker/usernameProvider` are framework hooks; users wire their own auth (the `learn/auth` lesson shows a recipe).

**Target:** Baked, opt-in auth providers + a uniform `Auth` facade:

```php
Auth::registerProvider(new OAuthProvider([
    'google' => ['client_id' => ..., 'client_secret' => ..., 'redirect' => ...],
    'github' => [...],
]));

Auth::registerProvider(new JwtProvider([
    'algo' => 'HS256',
    'secret' => getenv('JWT_SECRET'),
    'issuer' => 'zealphp.app',
]));

Auth::registerProvider(new CookieSessionProvider([
    'session_handler' => StoreSessionHandler::class,
]));

// Route protection middleware
$app->addMiddleware(new RequireAuthMiddleware(['routes' => ['/admin/*']]));

// In handlers — uniform user object regardless of provider:
$user = Auth::current();  // ?AuthUser with id, email, roles, providerName, raw token
```

Providers are pluggable; the framework ships three: OAuth2 (with PKCE), JWT, and session-cookie. Each registers its own login/callback routes under a configurable prefix. `Auth::current()` returns the same shape across providers so user code is provider-agnostic. ~600 LOC + tests + `learn/auth-oauth` lesson + portfolio page revamp.

### P1.4 — Coroutine sync primitives ✅ SHIPPED (`bfab6f4`)

**State today:** Users call `OpenSwoole\Coroutine\WaitGroup` / `Channel` / `Barrier` directly. No framework helper.

**Target:** Two helpers that cover 90% of the use case:

```php
// Fork-join: run all in parallel, collect results in input order
[$users, $posts, $stats] = App::parallel([
    fn() => DB::query('users'),
    fn() => DB::query('posts'),
    fn() => fetchStats(),
]);

// Bounded fan-out with concurrency limit:
$results = App::parallelLimit($jobs, fn($j) => process($j), concurrency: 10);
```

Internally these use `Coroutine\WaitGroup` + `Coroutine\Channel` of size `concurrency`. ~50 LOC + tests + a paragraph in `/coroutines`.

### P1.5 — Cluster-wide cron scheduler

**State today:** `App::tick($ms, $fn)` is per-worker (fires on EVERY worker on EVERY node). Useless for "run this at 03:00 UTC across the cluster — exactly once".

**Target:** Leader-election-backed cron with crontab expression support:

```php
App::onSchedule('0 3 * * *', function () {
    // Runs ONCE per cluster, regardless of how many ZealPHP processes/nodes.
    // Backed by Redis SETNX lock (5-minute lease) — winner runs.
    cleanupExpiredRows();
});

App::onSchedule('@every 30s', function () { /* convenience expressions */ });
```

Cron parser (e.g. `dragonmantank/cron-expression`) computes next-fire time; a per-worker `tick(1000ms)` checks if it's our turn AND we hold the lock. ~200 LOC + tests + `learn/cron` lesson.

### P1.6 — CSRF middleware + form helpers

**State today:** Nothing. Users roll their own token-generate + token-verify.

**Target:** Standard double-submit-cookie CSRF + a Twig-ish helper for forms:

```php
// app.php
$app->addMiddleware(new CsrfMiddleware([
    'exempt' => ['/api/webhook/*', '/health/*'],
    'token_lifetime' => 3600,
]));

// In templates — emit the hidden input + meta tag:
<?= csrf_field() ?>                // <input type="hidden" name="_csrf" value="...">
<?= csrf_meta() ?>                 // <meta name="csrf-token" content="..."> for fetch()

// JS side already supported by htmx via the meta tag — no extra setup.
```

Cookie-bound token (server-side rotated per session), header check on every non-GET/HEAD/OPTIONS that comes via cookie auth. Stateless verification — no Store hit per request. ~150 LOC + tests + middleware listing update.

### P1.7 — Memcached handler across Cache/Session/Store

**State today:** Redis + Table are the only handlers. Lots of installs already have Memcached deployed and don't want to bring up Redis.

**Target:** A `MemcachedBackend` implementing `StoreBackend`, plus matching session handler:

```php
Store::defaultBackend(StoreBackendKind::Memcached, [
    'servers' => [['memcache-1', 11211], ['memcache-2', 11211]],
]);

// Same Store/Counter/Cache API works through Memcached.
// Pub/sub NOT supported (Memcached has no pubsub) — throws clear StoreException.
```

Implementation: `ext-memcached` client (no native coroutine client in OpenSwoole, but `Runtime::HOOK_TCP` covers it). Counter via `increment`/`decrement` (Memcached has them). Cache via standard get/set. Sessions via `Memcached::set` with TTL.

Limitations: no pub/sub, no Streams. Documented as "Cache+Store with size cap, no messaging" — perfect for a side-cache deployment. ~300 LOC + tests + handler docs.

Add `StoreBackendKind::Memcached` to the enum.

### P1.8 — Cache pipeline + Store pipeline for true wire-batching

**State today:** `RedisBackend::mget`/`mset` are pipelined (H3 hardening). Cache doesn't expose pipelining — every `Cache::set` is a round-trip even when called in a loop.

**Target:** `Cache::pipeline(function($p) { $p->set(...); $p->set(...); })`:

```php
$values = Cache::pipeline(function ($p) use ($keys) {
    foreach ($keys as $k) {
        $p->get($k);
    }
});  // single round-trip on Redis backend; sequential on Table (no penalty)
```

Same shape on `Store::pipeline()`. ~80 LOC + tests + perf bench.

### P1.9 — CGI native path — retire `Legacy\FastCgiClient`

**State today:** `src/Legacy/FastCgiClient.php` (~260 LOC) is a hand-rolled FastCGI client. OpenSwoole 22.x ships `OpenSwoole\Coroutine\FastCGI\Client` as a built-in.

**Target:** Replace the hand-rolled client with the native one. The cgiFcgi dispatch path becomes:

```php
$client = new \OpenSwoole\Coroutine\FastCGI\Client('127.0.0.1', 9000);
$response = $client->request($params, $stdinBody, $timeoutMs);
```

Drop ~260 LOC of FastCGI parser code. Verify httpoxy CVE behaviour (`HTTP_PROXY` strip), 60s timeout, stderr drain still hold. ~30 LOC of replacement + integration test against the existing `cgi-bin` recipes.

### P1.10 — `App::stats()` aggregate + `/healthz` default ✅ PART-SHIPPED (`bfab6f4` — `App::stats()` done; `/healthz` middleware + Prometheus exposition still TBD)

**State today:** `Store::stats()` exists per-worker (H5). No aggregated framework health surface.

**Target:** A unified stats endpoint shape that surfaces:
- Worker count + status (running, idle, recycling)
- HTTP request count + 4xx/5xx breakdown
- WS connection count (`getClientList` aggregate)
- Store pool acquires/timeouts/created
- Pub/sub reconnects + handler errors
- Counter values for app-defined gauges
- Process memory (peak + current)

```php
$app->addMiddleware(new HealthCheckMiddleware([
    '/healthz' => ['liveness'  => fn() => true],         // alive if reachable
    '/readyz'  => ['readiness' => fn() => Store::ping()], // ready when store works
    '/metrics' => ['format' => 'prometheus'],            // optional Prometheus exposition
]));
```

`/metrics` emits Prometheus exposition format directly — no `prom/client_php` dependency. ~200 LOC + tests + portfolio page section.

### P1.11 — `Http::request()` outbound HTTP wrapper ✅ SHIPPED (`d34b771`)

**State today:** Users call curl (hooked, yields fine) or bring Guzzle/Symfony HTTP Client.

**Target:** Light ergonomic wrapper for the common case:

```php
$res = Http::get('https://api.example.com/users', ['Accept' => 'application/json']);
// $res = HttpResponse{ status: 200, body: '...', json: array|null, headers: ... }

$res = Http::post('https://api.example.com/webhook', json: ['k' => 'v'], headers: [...]);

// Concurrency + timeout:
$results = Http::all([
    fn() => Http::get('https://a.example.com'),
    fn() => Http::get('https://b.example.com'),
], timeout: 5.0);
```

Built on top of `OpenSwoole\Coroutine\Http\Client` (yields naturally). NOT a Guzzle replacement; a one-call helper. ~150 LOC + tests + portfolio page.

### P1.12 — `App::onSignal(SIGHUP, $fn)` user signal hooks ✅ SHIPPED (`bfab6f4`)

**State today:** Server reacts to SIGTERM/SIGUSR1 for graceful reload; user signal hooks not surfaced.

**Target:** Register handlers for arbitrary signals:

```php
App::onSignal(SIGHUP, function () {
    Config::reload();
    // … re-read config files without restarting the server …
});

App::onSignal(SIGUSR2, function () {
    StatsDumper::flush();
});
```

Plumbs into `OpenSwoole\Process::signal()`. Master-only by default; opt-in to per-worker. ~50 LOC + tests + paragraph in `/deployment`.

## Phase 2 (v0.4.0+) — bigger pieces, separate cycle

### P2.1 — `Process\Pool` / sidecar process API ✅ SHIPPED EARLY (`323df8b` — `App::onProcess()` via `$server->addProcess()`; the full Pool-with-IPC variant stays Phase 2)

**Target:** `App::onProcess('background-worker', $callable, ['workers' => 2])` — long-lived sidecar processes that share the framework's lifecycle. Different from task workers (which are queue-style); these are "always-on companion processes" (log shippers, file watchers, scheduled-job runners, etc).

Built on `OpenSwoole\Process\Pool`. ~300 LOC + lesson.

### P2.2 — gRPC service helper

**Target:** First-class gRPC server support on top of OpenSwoole's HTTP/2:

```php
$app->grpc('/com.example.Service/Method', $protobufHandler);
```

Generated stub integration via `grpc-tools` workflow. ~500 LOC + an `examples/grpc-service` scaffold + a `learn/grpc` lesson.

### P2.3 — HTTP/2 server-push helper

**Target:** `Response::pushPromise($url, $headers)` for HTTP/2 server push. Surface OpenSwoole's existing HTTP/2 capability through a typed method.

### P2.4 — Native Mail wrapper

**Target:** `Mail::send(MailMessage)` with pluggable transports (SMTP / sendmail / SES). No dependency on Symfony Mailer / PHPMailer required for the common case; advanced templating users wire those.

### P2.5 — GraphQL helper

**Target:** A thin layer over webonyx/graphql-php — schema registration + execution context wiring (per-request loader cache, auth integration with `Auth::current()`).

## Out of scope — intentional

| Area | Why deferred |
|---|---|
| **PDO / ORM / DB layer** | Three-way design decision: ship an ORM (which?), ship a thin DB helper (compete with PDO?), or punt to userland (current state). Owner has flagged this for separate discussion BEFORE writing code. Coroutine-PDO is also blocked by OpenSwoole 22.x hook limitations. Defer until OpenSwoole 25.x lands with PDO hooks OR we commit to an opinionated DB story. |
| **TCP / UDP non-HTTP listeners** | Niche use case. Users wanting MQTT brokers / custom binary protocols can use OpenSwoole's `Server` directly; doesn't need a framework helper. |
| **OpenSwoole `Coroutine\MySQL` / `Coroutine\PostgreSQL`** | Driver coverage is spotty. PDO-via-hook (when it lands) is the better path. |
| **CGI's "warm worker pool"** | Mentioned in some early notes; superseded by FastCGI native path (P1.9). |
